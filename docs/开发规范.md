## 写在前面

因为是团队协作开发，所以请大家务必学习一下git的使用方法，不要出现代码直接通过网页的文件上传位置进行提交（通常会出现"Add file via upload"等字样）。

对于git使用，可以选择无脑的方式，如使用vscode、Jetbrains中的版本控制插件或者功能，或者类似Github desktop等专门的图形化软件，也可以使用命令行。如果操作不熟悉，请不要对主分支进行直接操作，避免代码被覆盖。不会的内容直接上网搜或者问AI都可以解决。

## 提交规范

- 除了在仓库项目初始化阶段外，严禁直接提交主分支（master）分支，同时严禁直接在主分支开发，开发时将远程拉取至本地后，请开新分支进行开发
- 在当前分支中开发完后提交PR(pull request)，群里摇人审核一下（审核步骤是否有必要保留需要商讨），确认无问题后再提交到主分支
- 在每次开发前，记得先git fetch后再git pull下来最新的内容再进行开发
- 不要提交临时文件、构建文件等，如果项目中有相应文件，请在.gitignore中添加相应文件/路径

## Commit规范

### Commit Tittle

所有提交的 Commit Tittle都需要简要地讲一下做了什么，不要出现类似以下类型的提交

```bash
git commit -m "update"
```

Commit Tittle的示例可以参考以下内容：

```bash
git commit -m "[docs]初始化仓库，添加开发规范文档“
```

### 可用的Commit Tag

```text
[feat] feature:新特性/新功能
[fix] 修复 bug
[adjust] 值调整/参数调整
[optimize] 优化
[refactor] 重构/存在较大改动/架构更新
[clean] 清理代码/仓库
[docs] 文档相关
[test] 测试相关
[chore] 杂项，如修改 .gitignore 
```

### Commit Message（可选）

如果某次修改涉及的变动范围比较大，或者是出现了一个比较难看出来的问题被解决了，抑或是修改的内容不易被理解，那么就需要Commit Message。Commit Message需要使用Markdown格式，其中与Commit Tittle之间空一行。

## 代码书写规范

### 关于AI辅助

虽然使用Cursor等Agent确实可以解决很多问题，但是Agent往往带来的就是帮你修改接口，改的代码面目全非，对于稍大一些的项目完全是不可控且是灾难性的，而且对于并发控制或者是异步刷新逻辑并不能生成得很好。因此，在开发过程中，严禁直接使用Agent，可以使用Agent进行一些相关的提问或者得到修改建议等，但请不要直接让Agent对文件进行修改，除非你能完全讲清楚Agent到底做了什么，逻辑是什么，部分并不常见的语法糖等是什么意思为什么要这么用，存在的异步刷新逻辑等是否正确。

### 代码规范

- 勤写注释！！！建议特别是对于提供给别人用的接口，在函数开头写清楚对应接口输入输出是什么，函数的作用是什么，格式可以采取如下格式

```dart
/**
 * @brief 这里写这个函数的作用
 * @a 这里写输入参数是什么东西
 * @return 这里写返回的是什么东西
 */
int get_a_example(int a){
    return a;
}
```

- 注意命名规范，至少要让变量名和函数名看一眼就大概能猜到是什么意思，尤其是函数名，名称可以很长，但请不要低可读性。变量名严禁出现"a""abc"等，函数名也类似。
- UpperCamelCase（大驼峰）用于类、枚举和扩展，lowerCamelCase（小驼峰）用于变量和函数，以及lowercase_with_underscores（小写下划线）用于库、文件和文件夹名称
- 代码格式化，使用dart format或者flutter format进行代码格式化，具体如何进行格式化可以上网搜索或者问AI，一种简单的方式为

```bash
dart format .
```
