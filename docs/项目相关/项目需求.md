# 软件工程需求分析

## 用户需求：

作为一个华中科技大学的学生，我们身处于一个7000亩的庞大校园中，坐拥36个高质量食堂，但是苦于行动范围的限制以及对摊位菜单的研究不足，只能品尝到宿舍附近食堂的廖廖几个菜品。在某个晌午还会因为懒虫上身而花更贵的价钱去点各种形形色色的外卖，校外外卖虽然便捷，但存在价格偏高、食品安全及健康隐患等问题。
    
为了解决这些痛点，我们计划开发一款专门服务于华中科技大学师生的校园综合服务App，该App将提供以下核心功能：

- 美食发现与智能推荐： 基于用户的历史消费数据和个人偏好，智能推荐食堂及菜品。
- 在线下单与支付： 用户可以浏览所有食堂菜单，在线下单并完成支付。
- 校园跑腿服务： 提供校内物品代购、代取、代送等跑腿服务，由认证学生跑腿员接单。
- 用户与账户管理： 支持用户注册、登录、个人信息管理、地址管理及账户余额管理。
- 评价与信誉体系： 建立用户与跑腿员之间的双向评价机制，保障服务质量与平台信誉。
- 跑腿员认证与管理： 为希望成为跑腿员的学生提供线上认证渠道，并由管理员进行审核与管理。

## 功能需求：

### 1. 用户注册与登录

- 执行者：普通学生用户
- 触发条件：用户首次使用系统
- 交互动作序列：
  - 用户输入学号、用户名、邮箱、手机号、密码等注册信息；
  - 系统验证唯一性、学号合理性并创建账号；
  - 注册成功后可登录系统，进入个人主页。

### 2. 用户信息与地址管理

- 执行者：注册用户
- 交互动作序列：
  - 用户查看或修改个人资料（昵称、头像、联系方式等）；
  - 用户可新增、修改、删除收货地址，并设置默认地址。

### 3. 跑腿下单与接单功能

- 执行者：普通用户（下单者）、跑腿员（接单者）
- 交互动作序列：
  - 普通用户创建跑腿订单（填写取货地、送达地、备注信息、报酬等）；
  - 跑腿员登录后查看可接订单；
  - 跑腿员接单后，系统将订单状态设为“配送中”；
  - 配送完成后，用户确认收货并给予评价。

### 4. 账户余额与支付模块

- 执行者：注册用户
- 交互动作序列：
  - 用户通过校园卡或第三方支付充值余额；
  - 在下单时扣除相应费用；
  - 系统将部分报酬划入跑腿员账户。

### 5. 跑腿员认证与管理

- 执行者：学生用户、系统管理员
- 交互动作序列：
  - 学生用户提交身份证号、健康证等资料申请成为跑腿员；
  - 管理员审核通过后，用户获得“跑腿员”角色；
  - 跑腿员可查看历史订单、信誉分、收入等信息。

### 6. 菜品浏览与智能推荐

- 执行者：所有登录用户
- 交互动作序列：
  - 用户浏览各食堂与摊位菜单；
  - 系统根据热门程度等进行加权推荐；
  - 用户可对菜品点赞、收藏或评价。

## 非功能需求：

- 订单状态的流转和账户余额的变动必须保证事务一致性，杜绝数据错乱。
- 采用前后端分离的微服务架构，各模块之间低耦合，便于独立开发、部署和扩展。
- 用户密码、支付信息等敏感数据在存储和传输过程中使用强加密算法，保证安全性。
- 系统应支持至少 1000 名学生同时在线访问，响应时间不超过 2 秒。
